# 开发文档

## 1. 文档目的

这份文档是给当前项目所有组员使用的开发说明。主要作用有两个：

- 统一大家对项目当前状态的理解
- 把接下来要做的开发任务拆清楚，方便后续自行分工

本文档先不做人员分配，只整理“现在有哪些任务必须做、先后顺序大致是什么、每一块做到什么程度算完成”。

---

## 2. 项目当前情况

### 2.1 当前技术路线

项目已经确定采用以下方案：

- 后端：Java Servlet
- 前端：JSP
- 运行容器：Tomcat
- 构建方式：Maven Wrapper
- 数据存储：JSON 为主，CSV 用于导出，TXT / LOG 用于日志
- 数据库：不使用数据库

### 2.2 当前架构形式

目前项目不是前后端分离，而是传统 Java Web 多页面架构：

- JSP 负责页面显示
- Servlet 负责接收请求和页面跳转
- Service 负责业务逻辑
- Repository 负责 JSON 文件读写
- Model 负责实体对象定义

### 2.3 目前已经完成的基础工作

当前已经完成的内容如下：

- Maven Web 工程已经搭好
- Tomcat 启动方式已经配置好
- 项目 README 已补充启动说明和目录结构说明
- 基础目录结构已经建立
- 主要数据目录已经建立
- 核心实体类已经建立
- 部分 Repository 已经建立
- 已生成一批测试用 JSON 数据

### 2.4 当前目录分层

项目目前主要结构如下：

```text
src/main/java/com/bupt/tarecruit/
├── controller
├── service
├── repository
├── model
└── util

src/main/webapp/
├── WEB-INF
├── jsp
├── css
└── js

data/
├── users
├── jobs
├── applications
└── notifications
```

## 3. 当前阶段目标

目前最重要的目标不是继续扩技术，而是把系统主流程做通。

当前应该优先实现的闭环是：

1. MO 发布岗位
2. TA 浏览岗位
3. TA 提交申请
4. MO 查看申请并审核
5. TA 查看申请结果和通知

只要这条链路完整跑通，项目就从“有骨架”变成“可演示系统”。

## 4. 当前总任务清单

下面是目前项目需要完成的全部开发任务。后续分工可以直接从这些模块中认领。

### 4.1 公共基础任务

这部分是所有功能开发的基础。

#### 需要完成的内容

- 统一 ID 规则
  - User ID
  - Job ID
  - Application ID
  - Notification ID
- 统一状态字段
  - Job 状态
  - Application 状态
  - Notification 已读状态
- 补全缺少的 Repository
  - NotificationRepository
  - 日志读写相关工具或类
- 补充通用工具类
  - 时间工具
  - ID 生成工具
  - CSV 导出工具
  - 日志写入工具
- 统一异常处理和页面提示方式
- 统一 JSON 文件命名规则

#### 完成标准

- 后续所有模块都能使用统一的数据格式
- 新功能开发时不再重复写文件读写逻辑

---

### 4.2 登录与权限任务

系统目前缺少真正的入口，这一块必须尽快补齐。

#### 需要完成的内容

- 登录页面
- LoginServlet
- 按用户 ID 和密码登录
- 登录成功后写入 Session
- 根据角色跳转不同首页
  - TA 首页
  - MO 首页
  - Admin 首页
- 退出登录
- 未登录访问拦截
- 角色权限限制

#### 完成标准

- 用户可以正常登录
- 不同角色进入不同页面
- 不同角色不能越权访问其他角色功能

---

### 4.3 TA 模块任务

TA 端是系统中最完整的一条业务线，内容最多。

#### TA001 个人资料管理

需要完成：

- 个人资料页面
- 查看已有资料
- 编辑并保存以下内容
  - 学号
  - 专业
  - 年级
  - 已修课程
  - 成绩或 GPA
  - 相关经历
  - 证书信息
- 保存到用户 JSON

完成标准：

- TA 可以保存和修改自己的基础申请资料

#### TA002 上传 CV

需要完成：

- 上传简历功能
- 文件保存到 `uploads/`
- 在用户信息中保存文件路径
- 文件格式和大小校验

完成标准：

- TA 可以上传简历，后续 MO 可以查看或读取路径

#### TA003 浏览岗位

需要完成：

- 岗位列表页面
- 读取全部可申请岗位
- 只展示开放中的岗位
- 展示岗位核心信息
  - 课程名
  - 岗位类型
  - 要求
  - 工作量
  - 截止日期

完成标准：

- TA 能清楚看到当前可申请岗位

#### TA004 查看申请状态

需要完成：

- 查看自己提交过的所有申请
- 展示申请时间、岗位、状态、反馈
- 区分不同审核状态

完成标准：

- TA 能看到自己的申请进度

#### TA005 申请陈述

需要完成：

- 在申请页面增加 statement 输入框
- statement 设为可选
- 限制字数
- 保存到 Application 中

完成标准：

- TA 可以在申请时填写针对岗位的补充说明

#### TA006 登录通知

需要完成：

- 登录后读取通知
- 首页展示通知
- 展示状态更新和反馈内容
- 支持已读 / 未读

完成标准：

- TA 登录后能第一时间看到申请状态变化

#### TA007 - TA009 筛选增强

需要完成：

- 按开始日期排序或筛选
- 按课程名搜索
- 按 MO 名称筛选

完成标准：

- TA 能更快找到合适岗位

说明：

这一组优先级低于主流程，可以放在核心功能完成之后做。

### 4.4 MO 模块任务

MO 端负责让招聘流程真正闭环。

#### MO001 发布岗位

需要完成：

- 发布岗位页面
- 发布岗位表单
- 保存岗位信息到 JSON
- 发布后岗位能出现在 TA 岗位列表里

字段建议包括：

- 课程名称
- 岗位类型
- 任职要求
- 每周工作量
- 截止时间
- 发布人

完成标准：

- MO 能成功发布岗位

#### MO002 查看申请人

需要完成：

- 查看自己岗位的申请列表
- 列表中展示申请人基本信息
- 能看到申请时间和当前状态
- 支持查看 TA 资料、CV、申请陈述

完成标准：

- MO 可以完整查看申请信息

#### MO003 审核申请

需要完成：

- 对申请进行通过 / 拒绝 / 面试中标记
- 可填写反馈
- 更新申请状态
- 生成通知给 TA
- 记录审核人和审核时间

完成标准：

- MO 做出审核动作后，TA 端能看到变化

#### MO004 编辑 / 下架岗位

需要完成：

- 编辑岗位信息
- 下架岗位
- 截止日期后自动关闭岗位
- 下架后 TA 端不再显示
- 保留已有申请记录
- 记录操作日志

完成标准：

- 岗位可维护，不是一次发布后就无法管理

---

### 4.5 Admin 模块任务

Admin 端负责全局管理和规则控制。

#### AD001 三门课上限限制

需要完成：

- 统计 TA 当前承担课程数量

完成标准：

- 系统能够执行“最多 3 门课程”的规则

#### AD002 查看未招满岗位并提醒

需要完成：

- 展示未招满岗位
- 显示岗位、发布时间、申请人数等信息
- 标记长期未招满岗位

完成标准：

- Admin 可以跟进招聘进度

#### AD003 导出 / 导入 TA 信息

需要完成：

- 导出全部 TA 信息为 CSV
- 支持按状态筛选导出
- 导出文件写入 `exports/`

完成标准：

- 管理员可以导出系统中的 TA 数据

#### AD004 查看系统操作记录

需要完成：

- 记录核心操作日志
- 管理员页面查看日志
- 支持导出日志

完成标准：

- 系统具备基础可追踪性

#### AD005 冻结异常账号

需要完成：

- 标记异常账号
- 冻结账号后禁止登录或操作
- 页面提示账号状态

完成标准：

- 系统具备最基础的异常账号处理能力

说明：

## 6. 当前建议的开发顺序

建议整个小组按下面顺序推进，而不是所有人同时乱开工。

### 第一步：先补齐公共基础

- 把通用工具、缺少的 Repository、状态值、ID 规则统一

### 第二步：先打通登录和角色入口

- 让 TA、MO、Admin 都能进入各自首页

### 第三步：完成主流程闭环

- MO 发布岗位
- TA 浏览并申请
- MO 审核
- TA 查看结果

### 第四步：补阶段二核心任务

- 陈述、通知、编辑岗位、三门课限制、CSV 导出

### 第五步：补测试、文档和演示

- 让系统从“能跑”变成“能交”

---

## 7. 开发时的基本约定

为了避免后面越写越乱，建议全组统一以下规则：

- 不要直接绕过 Repository 手动读写 JSON
- 不要把所有逻辑都写进 Servlet
- 页面只负责展示和提交，不负责复杂判断
- 业务规则尽量放在 Service
- 提交代码前至少先做本地基本测试
- 新增字段时同步更新对应 JSON 数据结构
- 页面文案和状态值尽量统一命名

---

## 8. 现阶段的完成标准

当前阶段不是追求“所有功能都做完”，而是追求以下结果：

- 项目能启动
- 登录可用
- 三种角色都能进入系统
- 主招聘流程完整跑通
- 阶段二重点功能至少完成主要部分
- 有可演示页面
- 有基础测试数据
- 有清楚的运行说明

做到这一步，项目就已经进入比较稳的可演示状态。

---

## 9. 最后说明

接下来分工时，建议不要只按“页面”切，也不要只按“前端/后端”机械切。更适合的是按完整模块认领，比如：

- 登录与权限模块
- TA 模块
- MO 模块
- Admin 模块
- 公共工具与测试模块

这样更容易把责任边界划清，也更方便后面联调。
